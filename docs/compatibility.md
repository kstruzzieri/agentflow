# Artifact compatibility

Agentflow separates archival proof verification from mutable working-state
ingestion. A proof pack is intended to be immutable, portable, and
self-contained. The remaining `.agent/` files are execution state and may be
rewritten or extended while work is active.

## Historical guarantee

`verify-proof` is the only historical cross-version guarantee. Agentflow 1.x
verifies proofs built by Agentflow 0.4.0 or later and proofs built by every
Agentflow 1.y release where y is less than or equal to the verifier's minor
version. The floor is enforced against the proof-pack **schema** version
recorded in the proof (0.4.0, the schema written by the Agentflow 0.4.0
release); release numbers and proof-pack schema versions are distinct
sequences that happen to coincide at that floor. This promise applies to the
installed CLI command and the MCP `verify_proof` tool that delegates to it.

Agentflow 2.0 may drop pre-1.0 proofs only after announcing that change under
the deprecation policy and shipping a documented migration tool. This bounds
the maintenance promise without making existing 1.x proofs disposable.

An older verifier presented with a valid proof from a newer schema must reject
it as a **newer schema** and tell the user to **upgrade**. It must not describe
that expected version mismatch as tampering or corruption. Integrity failures
remain distinct diagnostics.

## Working-state boundary

There is no historical cross-major promise for `verify-run`, plan loading,
receipt loading, aggregation, mutation commands, or other working-state
consumers. Each reader supports only the versions allowed by that artifact's
declared compatibility policy:

- `exact`: the recorded version must equal the supported version;
- `same_major`: the major must match and the recorded minor must not exceed the
  supported minor; or
- `none`: the consumer deliberately makes no schema-version claim.

The executable policy table lives with the artifact constants in
`src/agentflow/contracts.py`. Schema-contract tests require every declared
artifact to have one policy so new readers cannot acquire an accidental
compatibility promise.

Before a major Agentflow upgrade, run **build-proof before a major upgrade** and
retain the proof bundle. That moves evidence from mutable working state into the
historically supported format. Starting with the 1.0 line, `doctor` warns when
the root working state was written by an older major and points to this ritual.

## Schema evolution

Within a schema major, a minor version may add optional fields only. Readers
must ignore unknown optional fields. Removing a field, making an optional field
required, changing a field's type or meaning, or changing canonical
serialization requires a schema-major increment.

Load-bearing schemas are `PLAN`, `EXECUTION_CONTRACT`, `PROOF_PACK`,
`STEP_RUNS`, `COMMAND_RECEIPTS`, `FILE_RECEIPTS`, `VERIFICATION_RUNS`, and
`DRIFT_REPORT`. Auxiliary schemas retain independent versions and are not
implicitly frozen when a load-bearing schema reaches 1.0.

## Compatibility fixtures

The compatibility matrix contains:

- the preserved legacy fixture whose proof schema is 0.4.0 but tool metadata is
  0.3.0;
- an immutable fixture generated by the checksum-pinned released v0.4.0
  `agentflow.pyz`;
- a full-featured current proof covering capabilities, runtime/MCP metadata,
  review evidence, criterion receipts, amendments, and hunk attribution; and
- an aggregated current proof containing `aggregation.json` and namespaced
  worktree roots.

The matrix invokes only the promised `verify-proof` command surface and asserts
both acceptance and diagnostic class. A future 1.0 fixture joins the same table
after the soak gate passes.
